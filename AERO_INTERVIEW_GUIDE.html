Why Two Separate API Routes?

/api/process-user-queries handles the batch loop with 1-second pacing. /api/user-query handles a single query with full Firebase auth, provider dispatch, and result formatting. This separation lets you call either independently — the dashboard uses /api/user-query for on-demand single queries too.

"The architecture is a layered Next.js monolith with two API route tiers. The outer batch route handles sequencing and pacing, the inner route handles auth and provider dispatch. The ProviderManager is the heart — it takes a generic request object and fans it out to all configured providers simultaneously using Promise.allSettled, which is critical because it means one slow or failing provider never blocks the others. If ChatGPT times out, we still get the Gemini result."

⚡ 3. The Provider System — Deep Dive

This is the most technically interesting part of Aero. Know it cold.

BaseAPIProvider — The Abstract Contract

Every provider extends this abstract class. Three methods MUST be implemented:

abstract execute(request: any): Promise<APIResponse>;
abstract validateRequest(request: any): boolean;
abstract transformResponse(rawResponse: any): any;

The base class provides shared infrastructure for free: makeRequest() (fetch with AbortController timeout + detailed error body extraction), retryRequest() (exponential backoff: 2s → 4s per retry), and checkRateLimit() (TODO: Redis — currently a no-op pass-through).

🧒 Simple: BaseAPIProvider is like a job description. It says "you MUST be able to execute, validate, and transform." Each specific provider (ChatGPT, Gemini, DataForSEO) fills in the exact implementation. The shared tools (retry logic, timeouts, HTTP) come included for free.

ChatGPT Search Provider — How it Actually Works

How does the ChatGPT integration work? And why responses.create instead of chat.completions.create?

We use the new OpenAI Responses API (client.responses.create) — not the older Chat Completions API. This is specifically because the Responses API supports the web_search_preview tool natively.

response = await client.responses.create({
  model: 'gpt-4.1-mini',
  tools: [{ type: "web_search_preview" }],
  input: request.input,
  temperature: request.temperature,
});

Model Waterfall: We don't hard-code a single model. We define an ordered list and try each in sequence:

// Search-capable models (primary):
['gpt-4.1-mini', 'gpt-4.1', 'gpt-4o-mini-search-preview-2025-03-11', 'gpt-4o-search-preview-2025-03-11']

// Fallback (no web search):
['gpt-4.1-mini', 'gpt-4.1']

If a model returns an error (rate limit, model not found, etc.), we catch it and try the next. If ALL search models fail, we fall back to no-search models. This means Aero essentially never hard-crashes on a model deprecation.

Detecting if web search was actually used:

private didUseWebSearch(rawResponse: any): boolean {
  const output = Array.isArray(rawResponse?.output) ? rawResponse.output : [];
  return output.some((item: any) => item?.type === 'web_search_call');
}

We inspect the output array for any item with type === 'web_search_call'. This tells us definitively if the model actually performed a real-time web search, which we store as webSearchUsed: boolean.

Annotation Extraction (Citations):

private extractAnnotations(rawResponse: any): any[] {
  if (Array.isArray(rawResponse.annotations)) return rawResponse.annotations;
  // Responses API nests annotations deeper:
  const output = Array.isArray(rawResponse.output) ? rawResponse.output : [];
  return output.flatMap(item => {
    const content = Array.isArray(item.content) ? item.content : [];
    return content.flatMap(contentItem =>
      Array.isArray(contentItem.annotations) ? contentItem.annotations : []
    );
  });
}

The Responses API has a nested structure: output[].content[].annotations[]. We flatten this to extract all citation URLs, which appear in the dashboard's Citations tab.

Gemini Provider — Grounding vs. Regular Search

How does Gemini's web search work differently from ChatGPT's?

Gemini uses Google Search grounding — a completely different mechanism. Instead of a tool call model, you enable grounding by passing tools: [{ google_search: {} }] in the request body to the REST API.

fetch(`https://generativelanguage.googleapis.com/v1beta/models/${model}:generateContent?key=${apiKey}`, {
  method: 'POST',
  body: JSON.stringify({
    contents: request.contents,         // [{ parts: [{ text: '...' }] }]
    tools: [{ google_search: {} }],     // Enables grounding
    generationConfig: { temperature: 0.7, maxOutputTokens: 4096 }
  })
})

The response comes back with rich metadata: candidate.groundingMetadata contains groundingChunks (the actual source URLs used) and citationMetadata. This gives us verifiable sources for every Gemini response.

Gemini Model Waterfall: Read from env var GEMINI_MODELS, defaults to:

gemini-2.5-flash → gemini-2.0-flash → gemini-flash-latest

Same pattern — tries each, catches errors, moves to next. One env var change swaps the entire model priority without code changes.

Google AI Overview Provider — DataForSEO

Why DataForSEO instead of calling Google directly? And how does it actually work?

Why DataForSEO: Google has no public API for AI Overviews (the AI-generated summary at the top of search results). DataForSEO is a third-party SERP data provider that actually executes real Google searches and returns the full structured response — including AI Overview content when it appears.

The Request Payload:

POST https://api.dataforseo.com/v3/serp/google/organic/live/advanced
Authorization: Basic <base64(username:password)>
[{
  keyword: "best crm software 2026",
  location_code: 2840,          // United States
  language_code: "en",
  device: "desktop",
  os: "windows",
  depth: 10,                    // Top 10 results
  group_organic_results: true,
  load_async_ai_overview: true, // Critical: waits for AI Overview to load
  people_also_ask_click_depth: 4
}]

Parsing the Response — Item Types: DataForSEO returns a flat items[] array. We filter by item.type:

const organicResults    = items.filter(i => i.type === 'organic');
const aiOverviewItems   = items.filter(i => i.type === 'ai_overview');
const aiOverviewRefs    = items.filter(i => i.type === 'ai_overview_reference');
const peopleAlsoAsk     = items.filter(i => i.type === 'people_also_ask_element');
const relatedSearches   = items.filter(i => i.type === 'related_searches');

The AI Overview item has a markdown field with the actual AI-generated text. The ai_overview_reference items have the domain and url that the AI Overview cited. This is gold for brand monitoring — "was my domain cited in Google's AI Overview?"

Cost: DataForSEO charges per API call (roughly $0.0001/request). Their cost is returned in the response body and we surface it in our per-query cost tracking.

ProviderManager — The Orchestration Brain

How does the ProviderManager know what to send to each provider?

transformRequestForProvider() takes the single generic APIRequest and maps it to the provider-specific format. The prompt also gets the cost-control instructions injected:

const contextualPrompt = `${basePrompt}

IMPORTANT COST RULES:
- Prefer answering from the supplied query and context first.
- Use web search only when a fresh or verifiable external fact is truly needed.
- If web search is needed, keep it to a single targeted search.
- Do not browse broadly, do not fan out across many sources.`;

Then per provider:

case 'chatgptsearch': return { input: contextualPrompt, model: 'gpt-4.1-mini', temperature: 0.7 };

case 'google-gemini': return { contents: [{ parts: [{ text: contextualPrompt }] }],
                               tools: [{ google_search: {} }],
                               generationConfig: { temperature: 0.7, maxOutputTokens: 4096 } };

case 'google-ai-overview': return { keyword: request.prompt, location_code: 2840,
                                    load_async_ai_overview: true, depth: 10 };
                                    // NOTE: Uses raw prompt, NO context injection

Notice DataForSEO doesn't get the context injection — it's a SERP scraper, not an LLM. It just needs the keyword.

"The most elegant thing about the provider system is the model waterfall. We maintain an ordered list of model candidates — if gpt-4.1-mini returns a rate-limit error, we silently try gpt-4.1, then the search preview models, then fall back to non-search models. The same pattern runs for Gemini across its model versions. It means model deprecations and rate limits are handled gracefully in production without any code changes — you just update an env var."

🗄️ 4. The Firestore + Cloud Storage Hybrid

This is a genuinely interesting engineering problem. Firestore has a hard 1MB document size limit. AI responses from multiple providers across 50+ queries easily blow past that. Here's how we solved it.

Why does your storage system have two tiers? Isn't Firestore enough?

A single ChatGPT response is often 2,000–4,000 characters. Multiply that by 2 providers × 50 queries = easily 400KB–800KB in one document. Add competitor analytics, metadata, timestamps — you're over 1MB fast.

The detection logic:

export function exceedsFirestoreLimit(data: any): boolean {
  const serialized = JSON.stringify(data);
  const sizeInBytes = new Blob([serialized]).size;
  const FIRESTORE_LIMIT = 1048576; // 1MB
  const SAFETY_MARGIN = 0.8;       // Use only 80% for safety
  return sizeInBytes > (FIRESTORE_LIMIT * SAFETY_MARGIN); // >~820KB triggers Cloud Storage
}

When triggered: storeDocumentWithLargeData() runs. It takes the large field (e.g. queryProcessingResults), uploads it as a JSON blob to Firebase Storage at path v8userbrands/{brandId}/{timestamp}.json, and stores a storageReference object in the Firestore document instead. The reference contains the download URL, file size, storage path.

// Firestore doc after offload (tiny):
{
  companyName: "Nike",
  storageReferences: {
    queryProcessingResults: {
      storageId: "2026-05-10-abc123",
      storagePath: "v8userbrands/brand123/2026-05-10-abc123.json",
      downloadUrl: "https://firebasestorage.googleapis.com/...",
      size: 450000
    }
  }
}

// Reconstruction on read:
const { document } = await retrieveDocumentWithLargeData('v8userbrands', brandId, ['queryProcessingResults']);
// → fetches Firestore doc, sees storageReferences, fires fetch(downloadUrl), merges back

The Triple-Layer Fallback for Writes

When saving results, we try the best approach first and degrade gracefully:

1
Cloud Storage (preferred for large data): Serialize all results as JSON blob → upload to Firebase Storage → save URL reference to Firestore. Keeps up to 100 historical results.
2
Truncated Firestore (medium data): Trim each AI response to 3,000 chars, keep last 50 results, double-check JSON is under 800KB, save directly.
3
Minimal Firestore (last resort): If still too large, save only metadata (timestamp, session ID, hasContent boolean) — NO actual AI text. At least query run history is preserved.
🧒 Simple version:

Firestore is like a notebook with a 5-page limit. When a query result is too big for the notebook, we put the actual text in a filing cabinet (Firebase Storage) and write only the file cabinet drawer number in the notebook. When we need the data, we look up the drawer number and retrieve it.

Cloud Storage Retrieval with Retry + Exponential Backoff

let retryCount = 0;
const maxRetries = 3;
while (retryCount < maxRetries) {
  try {
    const { document: fullBrandData } = await retrieveDocumentWithLargeData(...);
    if (fullBrandData?.queryProcessingResults) { break; } // success
  } catch (storageError) {
    retryCount++;
    const delay = Math.pow(2, retryCount) * 1000; // 2s, 4s, 8s
    await new Promise(resolve => setTimeout(resolve, delay));
  }
}

If all 3 attempts fail, we log detailed diagnostics (brand ID, Firestore query count, storage reference path) and continue analytics with truncated Firestore data.

"Firestore's 1MB document limit is a real constraint when you're storing multi-provider AI responses across dozens of queries. We built a transparent two-tier system: exceedsFirestoreLimit() detects when we're approaching the limit, then storeDocumentWithLargeData() automatically offloads the heavy field to Firebase Storage as a JSON blob and stores the download URL back in Firestore. Reads transparently reconstruct the full document. We also have a 3-attempt exponential backoff retry on Cloud Storage reads — 2 seconds, 4 seconds, 8 seconds — with fallback to truncated Firestore data if all attempts fail."

📊 5. The Analytics Engine

Brand Visibility Score — How It's Calculated

What exactly is the "brand visibility score" and how do you calculate it?
// For each query result across all providers:
totalProvidersWithBrandMentions += analysis.totals.providersWithBrandMention;
totalProviders += Object.keys(analysis.results).length;

// Final score:
const brandVisibilityScore = totalProviders > 0
  ? (totalProvidersWithBrandMentions / totalProviders) * 100
  : 0;

It's the percentage of provider-query combinations where your brand was mentioned. If you run 10 queries across 2 providers = 20 data points. If your brand appears in 14 of them, visibility score = 70%. It's a normalized metric that accounts for varying numbers of queries and providers.

Provider Ranking Algorithm — Multi-Signal Sort

This is where it gets clever. We rank providers by which one best surfaces your brand:

// Sort order (priority):
// 1st: Brand mentions (higher = better)
// 2nd: Domain citations ratio (domainCitations / totalCitations)
// 3rd: Total citations

providerRankings.sort((a, b) => {
  if (a.brandMentions !== b.brandMentions) return b.brandMentions - a.brandMentions;

  // Float comparison with epsilon to avoid rounding bugs:
  if (Math.abs(a.domainCitationsRatio - b.domainCitationsRatio) > 0.001)
    return b.domainCitationsRatio - a.domainCitationsRatio;

  return b.totalCitations - a.totalCitations;
});

Tie handling: If providers have identical brand mentions AND domain citation ratios, they're displayed as a tie: "chatgpt & google". Only providers with at least 1 brand mention or domain citation are eligible to rank — providers with zero performance output are explicitly excluded.

Lifetime Analytics — Multi-Source Aggregation

Lifetime analytics aggregates data from two sources:

  1. Brand document (v8userbrands): Current session's queryProcessingResults array. Retrieved from Cloud Storage if needed.
  2. Historical queries (v8userqueries): Legacy format with aiResponses[] array. Converted to current format on the fly.
// Legacy format conversion (fault-tolerant):
const convertedResult = {
  date: query.processedAt?.toDate()?.toISOString() || query.createdAt,
  processingSessionId: `legacy_${query.id}`,
  results: {
    chatgpt: { response: aiResponses.find(r => r.provider === 'openai')?.response || '' },
    googleAI: { aiOverview: aiResponses.find(r => r.provider === 'gemini')?.response || '' }
  }
};

Every format mismatch is caught with optional chaining and fallbacks — the system never crashes on stale data.

React Query Caching Strategy

// Latest session analytics: fresh for 3 min, cached 5 min
useQuery({ queryKey: ['latestBrandAnalytics', brandId],
           staleTime: 3 * 60 * 1000, gcTime: 5 * 60 * 1000, refetchOnWindowFocus: false })

// Lifetime analytics: fresh for 5 min, cached 10 min
useQuery({ queryKey: ['lifetimeBrandAnalytics', brandId],
           staleTime: 5 * 60 * 1000, gcTime: 10 * 60 * 1000, refetchOnWindowFocus: false })

Save throttle: Lifetime analytics is expensive to save to Firestore. We throttle it using localStorage:

const lastSave = localStorage.getItem(`lastAnalyticsSave_${brandId}`);
const shouldSave = !lastSave || (now - parseInt(lastSave)) > 5 * 60 * 1000; // 5 minutes
if (shouldSave) {
  saveLifetimeAnalytics(result).then(() => {
    localStorage.setItem(`lastAnalyticsSave_${brandId}`, now.toString());
  });
}

Competitor Analytics — Market Share Calculation

// Per competitor, ranked by composite score:
score = (mentions * 0.4) + (visibility * 0.4) + (citations * 0.2)

// Market share (relative to all competitors):
metric.marketShare = totalMentions > 0
  ? Math.round((metric.mentions / totalMentions) * 100 * 10) / 10
  : 0; // 1 decimal place, e.g. 34.7%

Competitor data is saved to a separate v8competitoranalytics collection. Document ID: {brandId}_{processingSessionId}. The latest 10 sessions are kept for trend analysis.

🛒 6. Amazon Integration & Domain Intelligence

Domain Intelligence — The Two-Step Company Info Pipeline

How do you extract company information from just a domain name?

Step 1 — Scrape real website data: We use jsdom server-side to fetch the actual website HTML and extract <title>, <meta name="description">, Open Graph tags (og:site_name, og:description). This is our primary source — actual content beats LLM hallucination.

Step 2 — AI enrichment prompt: We generate a structured prompt that says "use this website data as PRIMARY source" if scraping worked, or "research this domain" if it failed. The prompt demands a strict JSON output:

{
  "companyName": "...",
  "shortDescription": "2-3 sentences",
  "productsAndServices": ["list", "of", "offerings"],
  "keywords": ["4-5", "relevant", "phrases"],
  "competitors": ["3-5", "competitor", "names"]
}

Step 3 — Robust parsing: The AI response might wrap JSON in markdown code blocks (```json ... ```). We strip the fences, parse the JSON, validate each field, and fall back to domain-derived defaults if parsing fails.

Provider priority: ChatGPT Search first (better at current web data), Gemini as fallback. Both fire in parallel via ProviderManager, we use the first successful result.

Amazon ASIN Integration

How does the Amazon product tracking work?

ASIN extraction: User provides an ASIN or full Amazon URL. We extract the 10-character ASIN with regex.

Product data scraping: We use the Decodo API (a third-party Amazon scraper) to fetch: title, brand, category breadcrumbs, price, rating, review count, bullet points, top review, sales rank, sales volume.

Competitor enrichment via Gemini: Once we have the product data, we call Gemini with a targeted prompt:

`Find direct Amazon competitors for this product using only amazon.${marketplaceDomain}.
Product: ASIN ${asin}, Title: ${title}, Brand: ${brand}...
Return ONLY a valid JSON array of 4-8 competitor brand names.`

The response is parsed with a regex-based JSON extractor that handles markdown-wrapped arrays.

amazonOnlySearch mode: When enabled, queries get modified:

// Query transformation:
const providerQuery = amazonOnlySearch
  ? `site:amazon.com ${query.query}`
  : query.query;

// System prompt injection:
const amazonSearchInstruction = amazonOnlySearch
  ? ` Search only amazon.com for this prompt. Prefer product listings,
      Amazon search result pages, Amazon reviews. Do not use non-Amazon sources.`
  : '';

This forces the LLM to act as an Amazon-specific monitor — essentially telling it to behave like a product search engine rather than a general knowledge assistant.

"For Amazon brands, we built a special mode where every query gets prefixed with site:amazon.com and the system prompt explicitly restricts the LLM to Amazon-only sources. We also pull full product context — title, ASIN, category, bullet points — via the Decodo scraping API and inject it into the query context, then use Gemini to automatically identify competitor products. It's essentially turning a general-purpose LLM into an Amazon market intelligence tool."

🔐 7. Auth System — Firebase + JWT Handling

How Authentication Works End-to-End

1
Frontend: User signs in via Firebase Auth (Google OAuth or email/password). Firebase SDK returns a signed Firebase ID Token (JWT).
2
API Request: Every protected API call sends Authorization: Bearer <firebase-id-token> in the request header.
3
Server: auth.verifyIdToken(token) via Firebase Admin SDK. This makes a real cryptographic verification — not just a decode. Returns decodedToken.uid.
4
Profile Fetch: getUserProfileServer(uid) hits v8userbrands to load the user's profile and credits balance.

The Dev-Mode JWT Fallback (Unsafe but Useful)

// When Firebase Admin SDK is NOT configured (local dev):
const token = authorization.split(' ')[1];
const parts = token.split('.');
const payload = JSON.parse(Buffer.from(parts[1], 'base64').toString('utf8'));
const uid = payload.user_id || payload.uid;
console.warn('⚠️ Using decoded token UID (UNVERIFIED):', uid);

JWTs have 3 base64 parts: header.payload.signature. In local dev, we skip Admin SDK and just base64-decode the payload to extract the UID. This is explicitly marked UNSAFE for production — the signature is never verified. Great for dev speed, dangerous in prod.

BrandContext — Client-Side Brand Selection

// Persists selected brand in localStorage (survives page refresh)
localStorage.setItem('selectedBrandId', brandId);

// Hydration-safe: returns null on server-side render to prevent mismatch
if (!isClient) return <BrandContext.Provider value={{ selectedBrand: null, ... }}>...

// Auto-select first brand if none stored
useEffect(() => {
  if (brands.length > 0 && !selectedBrandId) {
    setSelectedBrandIdState(brands[0].id);
    localStorage.setItem('selectedBrandId', brands[0].id);
  }
}, [brands, selectedBrandId, isClient]);

// Validate stored brand still exists (handles deleted brands)
if (!brands.find(b => b.id === selectedBrandId)) {
  // fallback to first brand, clear invalid storage
}

"Auth runs on Firebase ID tokens. The API route calls Firebase Admin's verifyIdToken() which does real cryptographic signature verification — not just a decode. We have a local dev fallback that base64-decodes the JWT payload to extract the UID, but it's deliberately marked as unsafe and only active when Admin SDK isn't configured. For the client, we have a BrandContext that persists the selected brand ID to localStorage and handles all the edge cases: auto-selecting the first brand, validating the stored brand still exists, and preventing Next.js hydration mismatches by returning a null state on the first server render."

🚀 8. Scaling — What Would Change at 10k Brands

The current design runs queries synchronously in a Next.js API route. What's the problem with that at scale?

Vercel's function timeout limit is 60 seconds (or 300s on Pro). Processing 50 queries × 1-second pacing + ~10s per query = easily 10 minutes. You hit the timeout wall long before you finish.

At 10,000 brands with 50 queries each, that's 500,000 AI API calls per processing run — impossible to handle synchronously in a single HTTP request lifecycle.

The Migration Path

1
Async Job Queue: Replace the synchronous API call with a job enqueue. Return { jobId } immediately to the client. Use Google Cloud Tasks, BullMQ + Redis, or Trigger.dev as the queue.
2
Background Workers: Workers pick jobs off the queue, process one brand's queries at a time. Worker crashes don't affect other jobs — failed jobs get retried automatically.
3
Progress Tracking: Write job status to Firestore (processing → partial → complete). Client polls or uses Firestore onSnapshot() real-time listener to update the progress bar.
4
Rate Limit Control: The 1-second setTimeout pacing in the current loop gets replaced by queue rate limiting — each worker processes at a controlled rate per API key, allowing multiple workers without hitting provider limits.
5
Redis Caching: The checkRateLimit() stub in BaseAPIProvider is a // TODO: Redis. At scale, you implement a sliding window rate limiter using Redis INCR + EXPIRE to enforce per-API-key request budgets.

Database Migration — When to Leave Firestore

Would you stick with Firestore at enterprise scale, or migrate?

Firestore is great up to millions of documents, but it doesn't do arbitrary queries well. Want to ask "show me all brands with visibility score > 60% in the last 30 days, grouped by industry"? Firestore can't do that efficiently — it requires composite indexes and gets expensive fast.

Migration path: Add PostgreSQL (via Supabase or Cloud SQL) for analytics data. Keep Firestore for auth, user profiles, and real-time features (it excels there). Run a sync job from Firestore to Postgres for analytics queries.

The structured BrandAnalyticsData interface we already have maps cleanly to a relational schema — the migration wouldn't require redesigning the data model, just the storage layer.

"The current bottleneck is that we process queries synchronously inside a Vercel API route, which has a 60-second timeout ceiling. The fix is moving to an async job queue — something like Google Cloud Tasks or BullMQ. The API route just enqueues the job and returns a job ID. A background worker picks it up, processes queries at its own pace with proper per-provider rate limiting through Redis, and writes progress back to Firestore. The client either polls or uses a Firestore real-time listener to show progress. The rate limiting in BaseAPIProvider already has a Redis stub — it's the next thing to actually implement."

⚖️ 9. Trade-offs & Design Decisions

Why did you build your own provider abstraction layer instead of using LangChain?

LangChain adds a lot of abstraction overhead and opinionated patterns that don't fit well when you need to handle non-LLM data sources (DataForSEO is a SERP API, not a model) alongside LLMs. Our BaseAPIProvider handles both uniformly. LangChain would have forced us to shoehorn DataForSEO into an LLM-shaped interface, or maintain two separate systems. The custom layer also gives us exact control over cost tracking, model fallback ordering, and response normalization — things LangChain abstracts away in ways that are hard to override.

Why Firestore over PostgreSQL from the start?

Firebase gives you Auth + Firestore + Storage in one SDK with a generous free tier. For an early-stage product where the schema is evolving (brand data shape changed frequently), Firestore's schemaless nature meant we never had to write migrations. The tradeoff is the 1MB document limit and limited query capabilities — which we've already started working around with the Cloud Storage hybrid and React Query caching.

Why Promise.allSettled instead of Promise.all?

Promise.all rejects the entire batch if any provider fails. Promise.allSettled waits for all and returns each result as either { status: 'fulfilled', value } or { status: 'rejected', reason }. This is critical for a multi-provider system — if ChatGPT returns a 429 rate limit error, we still want the Gemini result. We filter to status === 'fulfilled' and surface provider errors individually in the response.

Why is the credits system disabled?

Credits were designed as a per-query metering system (10 credits per query). The infrastructure is fully built — the API reads credits, the Firestore profile stores the balance, deduction logic exists. It's been disabled (const requiredCredits = 0) likely during a growth/onboarding phase where friction reduction mattered more than monetization. The system is designed to re-enable with a single constant change — all the guard rails remain in place.

Why use localStorage to throttle analytics saves?

Recalculating and saving lifetime analytics runs on every dashboard load via React Query. If every user opening the dashboard triggered a Firestore write, you'd generate enormous write traffic. The 5-minute localStorage throttle ensures at most one save per browser session per 5 minutes. It's not perfect (different browsers won't share throttle state), but it reduces write pressure by ~10x for typical usage patterns with minimal complexity overhead.

💡 10. Gotcha Questions — Know These Cold

🚨 These are the questions that trip people up. Read carefully.
What happens if the AI response contains competitor mentions — does that count against our brand score?
Brand analytics and competitor analytics are completely separate pipelines. calculateCumulativeAnalytics() counts mentions of YOUR brand. calculateCompetitorAnalytics() is a separate function that counts competitor mentions. They never interfere. The brand visibility score only goes up if YOUR brand name appears in the response.
If the Firestore document exceeds 1MB, do you lose data?
No — exceedsFirestoreLimit() triggers before the write attempt. We offload to Cloud Storage first. If Cloud Storage also fails, we fall through to a truncated save (3000 char responses, last 50 results), then to a minimal save (metadata only). All three tiers preserve at minimum the session history — you never silently lose a processing run.
Next.js middleware exists in the codebase but doesn't actually block unauthorized requests. Why?
This is a known architectural decision. Next.js Edge middleware can't run Firebase Admin SDK (it uses Node.js APIs incompatible with the V8 edge runtime). So middleware runs but only does soft checks. Real auth enforcement happens inside each API route via authenticateUser() which calls auth.verifyIdToken() with the full Admin SDK. Dashboard routes rely on the client-side ProtectedRoute wrapper component that checks AuthContext.
The process-user-queries route calls /api/user-query per query — that's an HTTP call to itself. Why not just call the function directly?
Architecture choice for separation of concerns and future flexibility. Keeping routes as independent HTTP endpoints means: (1) Each route can be independently deployed or scaled. (2) /api/user-query can be called by external clients, mobile apps, or third-party integrations independently. (3) Error boundaries are clean — a crash in one query doesn't take down the whole batch handler. The 1-second delay between calls is also a natural rate limiter at the HTTP boundary. The cost is slightly higher latency per query from the loopback HTTP overhead.
How does the session ID system work? What's a "processing session"?
Every time a user clicks "Run" on the dashboard, we generate a processingSessionId = Date.now().toString(). All query results from that single run share this ID. It's used to: (1) Identify and replace results from the same run (prevents duplicate data if user reruns). (2) Group queries into "latest session" vs. historical sessions for analytics comparison. (3) Create the competitor analytics document ID: {brandId}_{sessionId}.
Why is google-ai-overview behind an environment flag?
if (process.env.ENABLE_GOOGLE_AI_OVERVIEW === 'true') selectedProviders.push('google-ai-overview') — DataForSEO has real per-call costs and is slower than LLM providers (it has to actually execute a Google search). The flag lets you enable it only when needed (paid tier users, specific use cases) and keep it off by default to control costs in development and for basic tiers.
What would break first if you had 1000 concurrent users running queries?
In order: (1) OpenAI API rate limits — 429s would cascade. (2) Vercel function concurrency limits — too many simultaneous long-running functions. (3) Firestore write rate — 1 write/second per document limit means concurrent updates to the same brand doc would queue up. (4) Firebase Storage upload bandwidth. The fix sequence: Redis rate limiting → async queue → Firestore write batching → Storage CDN.

📚 11. Jargon Cheatsheet — If They Ask "What Is..."

Promise.allSettled()
Runs multiple promises in parallel; waits for ALL to finish regardless of failures. Returns each result as fulfilled or rejected — never rejects the whole batch.
AbortController
Browser/Node API to cancel fetch requests. We set a timeout, and if the API doesn't respond in time, we call controller.abort() to cancel mid-flight.
Exponential backoff
Retry delays that grow exponentially: 2s, 4s, 8s. Prevents hammering a failing API. Formula: Math.pow(2, attempt) * 1000ms.
Grounding metadata
Gemini's way of telling you which real web sources it actually used to generate an answer. Contains groundingChunks (URLs) and citationMetadata.
SERP
Search Engine Results Page. DataForSEO fetches a full SERP and returns every element (organic results, AI Overview, People Also Ask, etc.) as structured JSON.
staleTime (React Query)
How long cached data is considered "fresh." During staleTime, React Query serves cache without refetching. After it expires, background refetch triggers on next use.
gcTime (React Query)
"Garbage Collection Time" — how long inactive cache entries stay in memory before being deleted. Data can be stale but still in GC cache.
Hydration mismatch
When server-rendered HTML differs from client's first render. Causes React warnings. Fixed by rendering null/loading state server-side using isClient flag.
Firebase ID Token
A short-lived (1 hour) signed JWT issued by Firebase Auth after login. Used to authenticate server-side API requests. Verified cryptographically by Firebase Admin SDK.
Abstract class
A class you can't instantiate directly — it defines the shape (methods) that subclasses MUST implement. BaseAPIProvider is abstract; ChatGPTSearchProvider is concrete.
Provider waterfall / model cascade
Trying models in priority order, catching errors at each step, moving to the next. Graceful degradation without hard-coding one model.
web_search_preview tool
OpenAI Responses API feature that lets gpt-4.1-mini perform real-time web searches during inference. Evidence of use detected via output[].type === 'web_search_call'.
load_async_ai_overview
DataForSEO parameter that tells it to wait for Google's JavaScript-rendered AI Overview to fully load before returning. Without it, AI Overview content is often missing.
Firestore 1MB limit
Hard ceiling on a single Firestore document. We detect approach at 820KB (80% safety margin) and offload large fields to Firebase Storage JSON blobs.
storageReference
A pointer object stored in Firestore that points to actual data in Firebase Storage. Contains storageId, storagePath, downloadUrl, size.
brand visibility score
(providers with brand mentions / total providers × queries) × 100. Normalized percentage showing how consistently your brand appears in AI responses.
