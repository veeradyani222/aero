⏱️ Session Duration & Limits

Google's Official Limits (without compression)

From Google's docs:

  • Audio-only sessions: 15 minutes max
  • Audio+Video sessions: 2 minutes max

After these limits, the session terminates — connection drops.

How Bloom Extends Sessions to UNLIMITED

We enable context window compression in the Live API config:

contextWindowCompression: {
  slidingWindow: { targetTokens: 6000 },
  triggerTokens: 15000,
}

What this does:

  • When context hits 15,000 tokens, compression triggers
  • Gemini automatically summarizes older audio/video, compresses to 6,000 tokens
  • This frees up space for new conversation
  • Sessions can now run indefinitely

Trade-off: AI gradually "forgets" details from earlier in the conversation. Recent context stays fresh, but older details get compressed away.

🧠 How to Make Sessions Truly Indefinite (Without Forgetting)

Problem: Compression is lossy — details fade as conversations get longer. For therapeutic continuity, the AI needs to remember everything.

Solution 1: External Memory with RAG (Best)

  • After each turn, extract key facts ("User mentioned feeling anxious about work presentation")
  • Store in vector database (Pinecone, Weaviate, or PostgreSQL with pgvector extension)
  • Before each AI response, search vector DB for relevant memories and inject them into the prompt
  • Result: AI has "perfect recall" of important details, even from months ago

Solution 2: Hierarchical Memory (Already Have This!)

  • Bloom ALREADY does this for between-session memory (daily → weekly → monthly compression)
  • Could extend to within-session memory: every 5 minutes, extract key points and store separately
  • Load these "session checkpoints" alongside real-time context

Solution 3: Hybrid Context Window

System Prompt (always present)
+ Retrieved Memories (from vector DB, ~2K tokens)
+ Recent Context (last 10 turns, ~8K tokens)
+ Current Turn (user input)

This gives AI both long-term memory (retrieved) and short-term memory (recent turns).

Jargon: "RAG (Retrieval-Augmented Generation) with semantic search. We use embeddings to retrieve contextually relevant memories and inject them into the prompt, giving the AI perfect recall without blowing up the context window."

🧒 Simple: Imagine talking to someone with a notebook. Normal compression = they erase old pages when the notebook fills up. RAG = they keep ALL notes in a filing cabinet, and when you mention something, they quickly pull out relevant old pages. They remember everything important, even from years ago!

"Right now, context compression extends sessions indefinitely, but it's lossy — the AI gradually forgets older details. To achieve true perfect recall, I'd implement RAG with a vector database. After each turn, extract key facts and store them as embeddings in something like Pinecone or pgvector (PostgreSQL extension). Then, before each AI response, do a semantic search to retrieve relevant memories and inject them into the prompt. This gives the AI both long-term memory (from the vector DB) and short-term memory (from recent context). We already have hierarchical compression for between-session memory — extending that to within-session checkpoints would be another option. The beauty of RAG is you get indefinite sessions WITHOUT forgetting anything important."

Video Call Optimizations (The Smart Stuff)

Video burns tokens WAY faster than audio (that's why the limit is 2 mins vs 15 mins). Bloom does several things to extend video sessions:

1. Low Frame Rate: 1 frame every 4 seconds

FRAME_CAPTURE_INTERVAL_MS = 4000; // 0.25 FPS, not 1 FPS

Google's docs say "up to 1 FPS" — we use 0.25 FPS to save tokens. The AI doesn't need to see you every second, just occasionally.

2. Low Resolution: 320×240 pixels

CAPTURE_WIDTH = 320;
CAPTURE_HEIGHT = 240;

Tiny frames = fewer tokens. Just enough for AI to see your face, not 4K detail.

3. JPEG Compression at 50% quality

canvas.toDataURL('image/jpeg', 0.5);

Compress the image before sending. Smaller file = fewer tokens.

4. Skip frames while AI is talking

if (modelTurnActiveRef.current) return; // Skip frames while AI responds

Why send video of you listening? We only capture frames when YOU'RE talking. Comment says this saves ~60% of video tokens.

Session Resumption (Handles Connection Drops)

WebSocket connections can drop (network issues, Google's 10-min connection limit). Bloom handles this:

sessionResumption: {
  handle: this._resumptionHandle || undefined,
}
  • Gemini sends us a resumption handle (like a save point)
  • If connection drops, we reconnect with that handle
  • Session continues where it left off — user doesn't notice
  • Handles are valid for 2 hours after disconnection

"For session duration — Google's Live API limits audio-only to 15 minutes and video to just 2 minutes without compression. We enable context window compression, which lets Gemini automatically summarize older context when it fills up, extending sessions to unlimited duration. For video specifically, we optimize aggressively: 1 frame every 4 seconds (not the default 1 FPS), 320×240 resolution, 50% JPEG compression, and we skip frames entirely while the AI is speaking — that alone saves about 60% of video tokens. We also use session resumption handles so if the WebSocket drops, we can reconnect without losing the conversation."

🧒 Simple version:

Without our optimizations, video calls die after 2 minutes. We fixed this by: (1) telling Gemini "summarize old stuff when memory gets full" — now calls can go forever, (2) sending tiny, compressed photos of you instead of HD video, (3) only sending photos when you're talking, not when AI is talking. It's like sending a postcard instead of streaming Netflix.

"Currently it's architected for early-stage scale — a few hundred concurrent users. The voice streaming itself scales well because it goes through Gemini's infrastructure, not ours. The bottlenecks are in-memory session state and synchronous analysis. To hit thousands of users, I'd add Redis for session persistence and move analysis to an async pipeline."

Scaling to 1 Million Users

Here's exactly what changes to handle massive scale:

1. Redis for TTS Cache

Problem: TTS voice preview cache lives in memory (Map object). Can't share across instances, lost on restart.

Solution: Redis Cluster (or AWS ElastiCache / GCP Memorystore). Cache TTS audio with TTL (e.g., 1 hour). Now any Cloud Run instance can serve cached audio. Saves repeated calls to Gemini's TTS API.

Jargon: "Externalize application cache to a distributed cache for horizontal scalability and cost reduction."

🧒 Simple: Right now, cached audio is stuck in ONE server's memory. Redis = shared storage that ALL servers can access. Server dies? Cache survives. User hits different server? Same cache.
2. Async Analysis + Transcript Writes → Pub/Sub + Cloud Functions

Problem: Post-call analysis blocks the API response for 3-5 seconds. ALSO, transcript writes during calls create connection pool pressure.

Solution: Use Google Pub/Sub for BOTH:

  • Transcript writes: During call, publish messages to queue → background worker batch-writes to DB
  • Analysis: On call end, publish to queue → Cloud Function runs analysis → writes results → notifies user

Add Dead Letter Queue (DLQ) for failed messages.

Jargon: "Event-driven architecture with message queue decoupling. Fire-and-forget with guaranteed delivery and async processing."

🧒 Simple: Instead of blocking to save things immediately, drop notes in a "to-do box". Background workers pick up notes and process them. User never waits. Like dropping mail in a mailbox — you don't wait for the postman.
3. Monolith → Microservices

Problem: Single Express server handles everything. Can't scale auth independently from analysis.

Solution: Split into 4 services:

  • Auth Service — handles login, JWT issuance, role switching
  • Call Service — session creation, transcript storage
  • Analysis Service — Gemini calls, dashboard insights (scale this the most)
  • API Gateway — routing, rate limiting, authentication

Jargon: "Service decomposition with domain boundaries. Independent deployment and scaling."

🧒 Simple: Instead of one person doing cooking + cleaning + serving, hire specialists. Need more cooking power? Hire more cooks without hiring more servers.
4. PostgreSQL → Read Replicas + Partitioning + Better Pooling

Problem: Single database becomes bottleneck. Neon's serverless tier has ~100 concurrent connections. Transcript writes exhaust the pool.

Solution:

  • Read replicas for dashboard queries (read-heavy workload)
  • PgBouncer for connection pooling with higher limits
  • Partition call_messages by month — queries only scan relevant partition, can drop old partitions
  • Separate connection pools for writes vs reads

Jargon: "Read/write splitting with connection pooling. Table partitioning by range for query performance and data lifecycle management."

🧒 Simple: ONE cashier for 1M customers = long line. Solution: 1 cashier for payments (writes), 5 cashiers for questions (reads). Partitioning = organize files by month instead of one giant pile — finding stuff is way faster.
5. Add CDN + Edge Caching

Problem: Static assets and repeated API responses hit origin servers.

Solution: Cloud CDN in front of Cloud Run. Cache static assets at edge. Cache dashboard API responses with Cache-Control headers (invalidate on call completion). TTS audio files go to Cloud Storage + CDN.

Jargon: "Edge caching with cache invalidation strategy. Reduce origin load by 80%."

🧒 Simple: Server in USA, user in India = slow. CDN = mini-copies in every country. User gets data from nearest location.
6. Rate Limiting + API Gateway

Problem: No protection against abuse. Gemini API costs explode.

Solution: Google Cloud API Gateway or Kong. Rate limit per user: 10 calls/day, 60 API requests/minute. Add quota management — free tier vs. paid tier. Circuit breaker pattern for Gemini failures.

Jargon: "API gateway with tiered rate limiting. Circuit breaker for fault tolerance."

🧒 Simple: Bouncer at the door. "You can only make 60 requests per minute. Come back later." Stops attackers and buggy apps from killing your servers.
7. Observability Stack → OpenTelemetry + Grafana

Problem: console.log doesn't scale. At 1M users, when someone reports "my call analysis is slow," you have ZERO visibility into which part is slow (DB fetch? Gemini API? Network?).

Solution: Implement the Three Pillars of Observability using OpenTelemetry + Grafana Cloud:

  • Industry standard — CNCF project, backed by Google, Microsoft, Grafana Labs, etc.

🎯 Why Grafana?

  • Fully supports OpenTelemetry — native ingestion via Tempo, Loki, Mimir
  • All-in-one platform — traces, logs, metrics in single UI
  • Powerful querying — PromQL for metrics, LogQL for logs, TraceQL for traces
  • Built-in alerting — Slack notifications, PagerDuty integration
  • Free tier — Grafana Cloud has generous free tier for testing

How Conversations Are Stored

The Transcript Storage Flow

Important: Audio/video streams NEVER touch the backend. They go directly to Gemini Live API. What we store are the text transcripts that Gemini sends back.

Step-by-step:

1
Call starts — Frontend calls POST /api/calls/start, backend creates row in call_sessions table, returns callId
2
During call — Gemini sends transcripts (user speech + AI responses). Frontend accumulates them in memory in a transcriptRef array
3
After every AI turn — Frontend calls POST /api/calls/:callId/messages with new messages. Backend does bulk insert into call_messages table
4
Call ends — Frontend calls PUT /api/calls/:callId/end, backend updates ended_at timestamp, triggers post-call analysis

Database schema:

call_sessions (user_id, call_type, started_at, ended_at)
call_messages (call_session_id, role, content, created_at)

Why save during the call? If browser crashes or connection drops, we don't lose the entire transcript — we have everything up to the last completed turn.

Authentication Flow

OAuth 2.0 + JWT with Role-Based Claims

User signs in with Google → Backend verifies the ID token using google-auth-library → We issue our OWN JWT (7-day expiry) containing userId, authRole (mom/therapist/trusted), targetUserId (if support role), and keyVersion (for revocation). This JWT is sent as Bearer token on every request.

🔥 3. What Was Actually Difficult (1-2 mins)

This is where you show depth. Pick ONE or TWO and go deep.

Challenge #1: Real-Time Barge-In Handling HARDEST

"The hardest part was handling barge-in — when the user starts speaking while the AI is mid-sentence. The server sends an 'interrupted' event via WebSocket, and we need to immediately stop the AudioStreamer, discard any buffered PCM chunks, and prepare for the new response. The tricky part is a race condition: if you set an 'ignore audio' flag while stale audio is still draining through the buffer, you might never get a turnComplete event to clear it — which permanently silences all future audio. We solved this by tracking whether the model is actively generating using a ref, and only setting the ignore flag when modelTurnActiveRef.current is true."

AudioWorklet

A low-latency audio processing API that runs on a separate thread from the main JavaScript thread. We use it to capture mic input at 16kHz without blocking the UI. Unlike the older ScriptProcessorNode, AudioWorklet doesn't cause audio glitches under load.

PCM (Pulse Code Modulation)

Raw, uncompressed audio format. We stream 16-bit PCM at 16kHz sample rate. Each frame is raw waveform data — no MP3/AAC compression. This is what Gemini expects and returns.

VAD (Voice Activity Detection)

Server-side detection of when the user is speaking. Gemini's Live API has built-in VAD that triggers the 'interrupted' event when it detects user speech during AI output.

Challenge #2: Post-Call Analysis Pipeline COMPLEX

"After each call ends, we run a multi-step analysis pipeline. First, we fetch the full transcript from the database. Then we call Gemini 2.5 Pro with a structured JSON schema — we enforce 8 signal scores (mood, energy, sleep, stress, support, self-kindness, coping, bonding), thematic analysis, risk assessment, and role-specific narratives for the mom, therapist, and trusted person. The challenge was resilience: Gemini API can fail, so we implemented model rotation — try Pro, fall back to Flash, fall back to Pro again — with exponential backoff (700ms × 2^attempt + jitter). We also clamp all scores to 0-100 and sanitize excerpts to prevent injection."

Structured Output / JSON Schema Enforcement

Instead of hoping the LLM returns valid JSON, we pass a JSON schema to the API and it GUARANTEES the response matches that structure. No more "oops the AI returned markdown instead of JSON" errors.

Exponential Backoff with Jitter

When retrying failed requests, wait exponentially longer each time (700ms, 1400ms, 2800ms...) plus a random offset (jitter). The jitter prevents "thundering herd" — all clients retrying at exactly the same time and overwhelming the server again.

Challenge #3: 3D Avatar Lip-Sync & Gestures COMPLEX

"The 3D avatars are built with Three.js using the TalkingHead library, which wraps Three.js scene management and provides an Oculus viseme morphing system. Here's the challenge: TalkingHead expects pre-computed word timings and viseme data from structured audio files. But Gemini Live API gives us raw streaming PCM audio at 24kHz with just text transcripts — no timing correlation, no viseme data. Real-time audio-to-viseme extraction would require expensive FFT analysis. So we built a procedural lip-sync system that injects random viseme animations directly into TalkingHead's animation queue every 1.8 seconds. We clear old visemes before pushing new ones to prevent queue flooding, and use a 1-second debounce before stopping to avoid premature gesture firing. For gestures, we built a rule-based mapper that pattern-matches transcript text — 'hello' triggers a wave gesture, 'sorry' shows a sad emoji. Moods apply immediately during speech, but gestures are buffered and fired only after speaking completes to avoid animation conflicts. It gives the illusion of perfect lip-sync with zero latency and no expensive audio processing."

Three.js

Popular 3D graphics library for the web. Renders 3D scenes using WebGL. We use it to render the avatar's face, apply lighting, and handle camera controls (OrbitControls for rotation, different positions for mobile vs desktop).

Viseme

Visual representation of a phoneme — the mouth shape for a speech sound. Oculus standard set includes 'aa', 'O', 'E', 'PP', 'FF', etc. TalkingHead animates the avatar's lips by morphing between these 14 viseme shapes.

GLTF/GLB

GL Transmission Format — standard 3D model format. GLB is the binary version. Our avatars are GLB files loaded with GLTFLoader and DRACOLoader for mesh compression.

The Lip-Sync Problem in Detail:

What TalkingHead Expects:

{
  audio: AudioBuffer,
  words: ['Hello', 'how', 'are', 'you'],
  wtimes: [0, 0.3, 0.6, 0.9],  // word start times
  wdurations: [0.25, 0.25, 0.25, 0.3],
  visemes: ['HH', 'AH', 'L', 'OW', ...],  // optional
  vtimes: [0, 0.05, 0.1, 0.15, ...],
  vdurations: [0.04, 0.04, 0.05, 0.06, ...]
}

What Gemini Gives Us:

{
  mimeType: 'audio/pcm;rate=24000',
  data: 'base64EncodedPCMChunk...',
  // NO word timings, NO viseme data, just raw audio stream
}
// Separate event:
{ transcript: 'Hello how are you' }  // No timing correlation!

❌ Why We Can't Match Them:

  • Audio arrives as chunks every ~100ms, transcript arrives separately
  • No timestamp linking audio chunk to transcript words
  • Extracting visemes from raw PCM requires FFT analysis + phoneme recognition (expensive, high latency)
  • Would need to buffer audio to analyze, killing real-time experience

The Clever Solution: Procedural Lip-Sync

Instead of analyzing audio, generate random viseme animations:

// VideoCallPage.jsx - Procedural lip-sync system
const visemes = ['aa', 'O', 'E', 'I', 'U', 'PP', 'FF', 'TH', 
                 'DD', 'kk', 'CH', 'SS', 'nn', 'RR'];

const pushVisemes = () => {
  const h = headRef.current;
  if (!h || !h.animQueue) return;

  // 1. Clear old visemes (prevent queue flooding)
  h.animQueue = h.animQueue.filter(x => x.template?.name !== 'viseme');

  // 2. Push ~1 second of random viseme animations
  const now = h.animClock || 0;
  for (let i = 0; i < 5; i++) {
    const v = visemes[Math.floor(Math.random() * visemes.length)];
    const t = now + i * 200;  // 200ms apart
    const d = 180;  // 180ms duration
    h.animQueue.push({
      template: { name: 'viseme' },
      ts: [t - d/3, t + d/2, t + d + d/2],  // Animation curve
      vs: {
        ['viseme_' + v]: [null, 0.5 + Math.random() * 0.25, 0]
      }
    });
  }
};

// 3. Run every 1.8 seconds while AI is speaking
if (modelTurnActive) {
  proceduralIntervalRef.current = setInterval(pushVisemes, 1800);
}

✅ Why This Works:

  • Zero latency — no audio analysis, instant animation
  • Visually convincing — random variety looks natural with audio playback
  • Lightweight — just random selection + queue manipulation
  • Language-agnostic — works for any language equally
  • No false sync — better to have plausible randomness than visibly wrong sync

Gesture System: Rule-Based Pattern Matching

Pattern matching on transcript text triggers gestures and moods:

// gestureMapper.js - Rule-based gesture mapping
function mapSentence(text) {
  const t = text.toLowerCase();
  
  if (/\b(hi|hello|hey|greetings)\b/.test(t))
    return ['gesture:handup', 'mood:happy'];
    
  if (/\b(sorry|unfortunate|sad)\b/.test(t))
    return ['emoji:🥺', 'mood:sad'];
    
  if (/\b(sure|yes|yeah|agree)\b/.test(t))
    return ['nod', 'emoji:😉'];
    
  if (/\b(no|nope|disagree)\b/.test(t))
    return ['shake', 'emoji:😶'];
    
  return ['nod'];  // default
}

⏱️ Timing Strategy: Deferred Execution

// Apply moods immediately, buffer gestures for later
function scheduleActions(actions) {
  for (const a of actions) {
    if (a.startsWith('mood:')) {
      exec(a);  // Safe during speech - just changes expression
    } else {
      pendingActions.push(a);  // Buffer gesture/emoji
    }
  }
}

// After speech completes, fire buffered gestures
function flushAfterSpeaking() {
  const toFire = pendingActions.slice(-2);  // Last 2 to avoid spam
  toFire.forEach(exec);
  pendingActions = [];
}

🎯 Available Actions:

  • Gestures: nod, shake, handup, thumbup, namaste
  • Moods: happy, sad, neutral, angry, surprised
  • Emojis: 🤣, 🥰, 🥺, 😉, 😶, 😇, etc.
  • Cooldowns: 1.5s for nod, 2.5s for shake, 3s for gestures, 4s for moods

Challenge #4: Hierarchical Memory Compression NOVEL

"We needed the AI to remember past conversations without blowing up token costs or latency. Instead of using a vector database with embeddings, we built a hierarchical compression system. After each call, Gemini extracts ONE insight (max 28 words) and upserts it into Level 0 (daily). When Level 0 hits 10 entries, we compress them into a single Level 1 entry (max 45 words). Level 1 compresses to Level 2 the same way. This gives us bounded storage — max ~30 rows per user — and bounded tokens — about 10 sentences in the system prompt. It's simpler infrastructure than Pinecone or Weaviate, and compression actually surfaces recurring patterns better than raw retrieval."

Vector Database / Embeddings

The "standard" approach: convert text to high-dimensional vectors using an embedding model, store in a specialized database (Pinecone, Weaviate, Chroma), and do similarity search. We avoided this complexity.

Upsert Pattern

INSERT if the row doesn't exist, UPDATE if it does — in a single atomic operation. We use PostgreSQL's ON CONFLICT DO UPDATE to merge today's new insight with any existing one.

⚖️ 4. Trade-Offs You Made

Interviewers LOVE when you acknowledge trade-offs. It shows maturity.

Trade-off #1: JWT in localStorage vs HttpOnly Cookies

"We store JWTs in localStorage for simplicity — no CSRF tokens needed, works cleanly with our SPA architecture. The trade-off is XSS vulnerability: if an attacker injects malicious JavaScript, they can steal the token. In production, I'd move to HttpOnly cookies with SameSite=Strict and add CSRF protection. We mitigated XSS risk by using React (auto-escapes output) and sanitizing all user-generated content."

Trade-off #2: JSONB vs Normalized Tables for Analysis Storage

"We store analysis results in a JSONB column rather than normalized tables. This gives us schema flexibility — we can add new signal types without migrations. The trade-off is query performance: JSONB requires full document parsing, and indexing nested fields needs GIN indexes. For our scale (hundreds of users), this is fine. At scale, I'd extract hot fields into indexed columns and keep JSONB for the long tail."

Trade-off #3: No Text Chat Feature (Voice/Video Only)

"We built infrastructure for text-based chat with the companion using Google ADK, but ultimately focused on voice and video only. Voice/video use Gemini Live API directly via WebSocket. The ADK code exists in the backend but isn't used in production — it was an early experiment. The trade-off was focus — polishing one interaction mode (real-time voice) rather than spreading thin across multiple modalities. Text chat would require different UX patterns and doesn't offer the emotional connection of hearing a voice."

Trade-off #4: Synchronous Analysis vs Async Queue

"Post-call analysis runs synchronously when the call ends. The user waits ~3-5 seconds for the API call to complete. The trade-off was simplicity vs responsiveness. In production, I'd push to an async queue (Cloud Tasks or Pub/Sub), return immediately, and notify the user when analysis is ready. This also helps with Gemini rate limits."

📈 5. How Would You Scale This?

The "at scale" question. Have 3 bullets ready.

"Three things I'd change at scale:"

1
Async Analysis Pipeline

Move post-call analysis to Pub/Sub + Cloud Functions. When a call ends, publish a message to a topic. A Cloud Function picks it up, runs analysis, and writes to the database. The call-end API returns immediately. This handles Gemini rate limits gracefully and improves UX.

2
Redis for Session State

Replace in-memory ADK runner cache with Redis. Sessions survive server restarts, work across Cloud Run instances, and can be TTL'd for automatic cleanup. This is critical for horizontal scaling.

3
Aggressive Caching

Add Redis caching for dashboard insights. Data only changes when a call completes, so cache aggressively with explicit invalidation. Also cache TTS audio files in Cloud CDN instead of in-memory.

Bonus Points: What Breaks at 10,000 Users?

  • Gemini Live API costs: 10K users × 10 min/call × $0.23/call = $2,300/day (voice/video)
  • Analysis costs: 10K × 2 calls/day × $0.002 = $40/day (negligible)
  • Database write spam: 10K concurrent calls × 30 turns each = 300K writes during peak, exhausts connection pool
  • Database growth: 10K × 2 calls/day × 50 messages = 1M rows/day → 30M rows/month
  • Connection pool exhaustion: Neon has ~100 concurrent connections, saving every turn requires connection per request
  • Single Express process: Would need to break into microservices (Auth, Call, Analysis, Dashboard)
  • TTS cache: In-memory Map can't share across instances (need Redis)

Deep Dive: The Transcript Storage Bottleneck

🔴 Critical Issue at Scale: Incremental transcript saving becomes a bottleneck.
Problem: Write Amplification

Current: Save transcript after EVERY AI turn (every 20-30 seconds)

  • 10-minute call = 20-30 turns = 20-30 API requests
  • 1,000 concurrent calls = 20,000-30,000 database writes per 10 minutes
  • 10K concurrent calls = 500-600 writes per SECOND

What breaks:

  • Connection pool exhaustion — each save grabs a connection, pool only has 100
  • API request overhead — network latency, JWT validation, connection wait
  • Table bloat — billions of rows after months of operation
Solution 1: Batch Saves (Easy Win)
// Instead of every turn
if (turnComplete) saveMessages();

// Save every 5 turns OR every 30 seconds
if (turnCount % 5 === 0 || timeSinceLastSave > 30000) {
  saveMessages();
}

Result: 80% fewer API calls during calls. Still safe — lose max 5 turns if crash.

Solution 2: Pub/Sub Queue (Best for Scale)

Instead of direct database writes, publish to a queue:

// Frontend → Backend → Pub/Sub
POST /api/calls/queue-messages
→ pubsub.publish('transcript-writes', { callId, messages })

// Background worker
Worker pulls from queue
→ Batches 100 messages
→ Single bulk INSERT

Benefits:

  • Non-blocking writes (no connection pool pressure)
  • Workers can batch writes intelligently
  • Handles traffic spikes gracefully
Solution 3: Table Partitioning

Partition call_messages by month:

CREATE TABLE call_messages_2026_04 
PARTITION OF call_messages
FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

Benefits:

  • Queries only scan relevant partition (faster)
  • Can drop old partitions easily (auto-delete old data)
  • Better index performance
Solution 4: localStorage Backup + Save at End

Don't save during call. Save everything at end:

// During call: save to localStorage
localStorage.setItem(`call-${callId}`, JSON.stringify(transcript));

// On call end: bulk save
PUT /api/calls/:callId/end
{ transcript: [...all messages] }

Trade-off: If browser crashes, transcript lost from server BUT recoverable from localStorage on next visit.

"The incremental transcript saving is great for reliability, but at scale it creates write amplification. With 10K concurrent calls, we'd be doing 500+ writes per second, exhausting the connection pool. I'd fix this by batching saves — save every 30 seconds instead of every turn, reducing API calls by 80%. Even better, use Pub/Sub with background workers — the frontend sends messages to a queue, workers batch-write them to the database. This eliminates blocking on database connections and handles traffic spikes gracefully. We'd also partition call_messages by month so queries stay fast even with billions of rows."

📚 6. Jargon Cheatsheet

Memorize these. Drop them naturally.

WebSocket
Persistent bidirectional connection. Unlike HTTP (request-response), WebSocket keeps the connection open for real-time streaming.
AudioWorklet
Web API for low-latency audio processing on a separate thread. Captures mic at 16kHz without blocking UI.
PCM (Pulse Code Modulation)
Raw, uncompressed audio. 16-bit samples at 16kHz. What Gemini expects for voice input/output.
VAD (Voice Activity Detection)
Detects when someone is speaking. Gemini uses this to trigger "interrupted" events for barge-in.
Barge-In
When the user interrupts the AI mid-sentence. Requires stopping playback and discarding buffered audio.
Structured Output
Force LLM to return JSON matching a schema. Guarantees parseable output, no "oops it returned markdown."
Exponential Backoff
Retry strategy: wait 1s, then 2s, then 4s, etc. Prevents hammering a failing service.
Jitter
Random offset added to retry delays. Prevents "thundering herd" — all clients retrying at once.
JSONB
PostgreSQL's binary JSON type. Queryable, indexable (GIN indexes), but slower than columns.
GIN Index
Generalized Inverted Index. Used for indexing JSONB fields and full-text search in PostgreSQL.
Upsert
INSERT if new, UPDATE if exists. PostgreSQL: ON CONFLICT DO UPDATE.
Connection Pooling
Reuse database connections instead of creating new ones. Neon + pg library handle this.
Stateless Backend
Server doesn't store session state. Each request is independent. Scales horizontally.
RBAC (Role-Based Access Control)
Permissions based on roles (mom, therapist, trusted). JWT contains role, middleware enforces it.
Key Versioning
Increment a version number to revoke all old tokens. JWT has version, DB has current version. Mismatch = rejected.
Viseme
Visual representation of a phoneme (lip shape). Used for lip-syncing the 3D avatar to speech.
ADK (Agent Development Kit)
Google's framework for building conversational AI agents with memory and tool use.
Context Window
Max tokens an LLM can process at once. We trigger compression at 15K tokens, target 6K.
RAG (Retrieval-Augmented Generation)
Technique where you search external memory (vector DB) for relevant info and inject it into the prompt. Gives AI "perfect recall."
Vector Database
Database that stores text as high-dimensional vectors (embeddings) for semantic similarity search. Examples: Pinecone, Weaviate, pgvector.
Embeddings
Numeric representation of text as a vector. Similar meanings = vectors close together in space. Used for semantic search.
Semantic Search
Search by meaning, not keywords. "I feel sad" matches "user experiencing depression" even without shared words.
OpenTelemetry
Open-source observability standard for traces, metrics, and logs. Vendor-neutral — works with Grafana, DataDog, etc.
Distributed Tracing
Following a single request through multiple services with a trace ID. See exactly where latency occurs.
Correlation ID / Trace ID
Unique ID attached to a request. Links logs, traces, and metrics for that specific request together.
Grafana
Observability platform for visualizing metrics, logs, and traces. Products: Tempo (traces), Loki (logs), Mimir (metrics).
Pub/Sub
Message queue pattern. Publish messages to a topic, subscribers process them asynchronously. Decouples services.
Dead Letter Queue (DLQ)
Queue for messages that failed processing after max retries. Prevents infinite loops and data loss.

❓ 7. Gotcha Questions They Might Ask

"Why not use a vector database for memory?"

"We DO use hierarchical compression for between-session memory, which works well for cross-call continuity. For within-session memory right now, we rely on Gemini's context window with compression. The trade-off is the AI gradually forgets older details in long sessions. If we needed true perfect recall for indefinite sessions, I'd absolutely add a vector database with RAG — store embeddings of each turn, then retrieve relevant memories semantically before each response. pgvector is appealing because it's just a PostgreSQL extension, no new infrastructure. The challenge is latency — every turn adds a vector search query, which could add 50-100ms. For our use case, recent context compression is 'good enough,' but for clinical continuity at scale, RAG would be the right choice."

"What happens if Gemini goes down?"

"For voice calls: the WebSocket disconnects, we show an error state, and offer to retry. We also have resumption handles — if the connection drops mid-call, we can resume from where we left off rather than losing the entire conversation. For analysis: we have model rotation (Pro → Flash → Pro) with exponential backoff. If all attempts fail, we mark the analysis as 'pending' and show the user their call completed but insights are delayed. The call itself isn't lost — transcripts are already persisted."

"How do you handle sensitive health data?"

"Three layers: First, access control — only the mom, her linked therapist, and trusted person can see her data, enforced via JWT role claims and middleware. Second, key rotation — the mom can instantly revoke all access by regenerating her share key. Third, data minimization — therapists see clinical summaries, not raw transcripts; trusted persons see plain-language suggestions, not analysis details. For HIPAA compliance at scale, we'd add encryption at rest, audit logging, and BAA agreements with Google Cloud."

"Why PostgreSQL instead of MongoDB?"

"We needed relational integrity — users link to companions, calls link to messages, support roles link two users. Foreign keys and JOIN queries are cleaner in PostgreSQL. We used JSONB for the analysis column to get document flexibility where we needed it, so we got the best of both worlds. Also, PostgreSQL on Neon gives us serverless scaling and connection pooling out of the box."

"What's your testing strategy?"

"For the backend: Zod schemas validate all inputs at the API boundary — that catches most issues. For complex logic like the analysis pipeline, we have integration tests that mock Gemini responses. For the frontend: the real challenge is the audio pipeline, which is hard to unit test. We rely on manual E2E testing and error monitoring in production. If I had more time, I'd add Playwright tests for critical flows and audio mocking for the call hooks."

"What would you do differently?"

"Three things: First, async analysis from day one — we added it synchronously for simplicity but it's now blocking UX. Second, TypeScript everywhere — we're using JavaScript with Zod, but full TypeScript would catch more errors at compile time. Third, better error boundaries — when Gemini fails, the user experience isn't graceful enough. I'd add more specific error states and retry UX."
